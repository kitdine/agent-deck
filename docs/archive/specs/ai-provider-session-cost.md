# Local AI Provider Session Cost Tracking Design

This document remains the detailed behavioral reference for the former legacy
usage implementation and fixtures. The repository implementation was removed
after the AgentDeck replacement passed independent review. The AgentDeck
specification is the active architecture and changes the privacy boundary for
the separate, opt-in session search database.

## Goal

Calculate local Codex and Claude session costs from their existing JSONL files.
Use official model prices, then apply the price multiplier configured for the
provider selected by `ai-provider-mode`. Keep all collection, normalization,
attribution, storage, and reporting local.

The task name is **Local AI Provider Session Cost Tracking**.

## Scope

The first version will:

- scan Codex and Claude JSONL files without reading usage from provider APIs;
- persist normalized usage in a mode-`0600` SQLite database and scan only new
  or changed data on later runs;
- calculate costs from a versioned snapshot of downloadable official model
  prices;
- apply one price multiplier per provider, defaulting to `1`;
- attribute resumed sessions precisely when they are started through a wrapper;
- provide a session-start estimate when the wrapper was not used;
- report raw usage, official base cost, multiplier, provider cost, and
  attribution quality.

The first version will not support custom model pricing, provider billing APIs,
invoice reconciliation, a background daemon, a graphical interface, or storage
of prompts and responses.

## Terminology

- **Logical session:** the durable Codex or Claude conversation identified by
  the client session ID. A logical session can be resumed more than once.
- **Run:** one process lifetime in which a new or resumed logical session is
  used. Different runs of the same logical session may use different providers.
- **Exact attribution:** a run launched through `ai-provider-run`, allowing its
  file changes to be bound to the provider active at launch.
- **Estimated attribution:** a logical session not observed through the
  wrapper. Its complete usage is assigned to the provider active when its first
  record was written.
- **Historical attribution:** a session for which no provider selection record
  exists. It belongs to the separate `historical` category with multiplier `1`.

## Local Inputs

The scanner reads only JSON objects and selected metadata fields from:

```text
~/.codex/sessions/**/*.jsonl
~/.codex/archived_sessions/*.jsonl
~/.claude/projects/**/*.jsonl
```

It extracts timestamps, logical session IDs, turn or response IDs, model IDs,
and token counts. It must not persist or print prompt text, response content,
tool arguments, credentials, images, attachments, or unrelated client state.

Codex usage is normalized from `event_msg` records with
`payload.type = "token_count"`. The scanner uses `last_token_usage` for the
current turn, not cumulative `total_token_usage`. `cached_input_tokens` is a
subset of `input_tokens`; `reasoning_output_tokens` is diagnostic metadata and
must not be added to `output_tokens` a second time.

Claude usage is normalized from assistant message usage. Repeated snapshots of
the same message ID are upserted rather than summed. Synthetic messages and
zero-usage placeholders do not create billable events. Claude cache creation
must retain both fields when present:

```text
cache_creation.ephemeral_5m_input_tokens
cache_creation.ephemeral_1h_input_tokens
```

The aggregate `cache_creation_input_tokens` is retained for diagnostics but is
not added to the two detailed fields. If a record has aggregate cache creation
tokens but no TTL breakdown, those tokens remain visible and their cache-write
cost is marked unpriced instead of guessing a TTL.

## Provider Configuration

Each provider may define one non-negative price multiplier:

```json
{
  "providers": {
    "official": {
      "host": "http://10.0.0.103:15021",
      "cost_multiplier": 1,
      "codex": { "auth": "openai_login" },
      "claude": { "enabled": false }
    },
    "aigocode": {
      "host": "http://10.0.0.103:15022",
      "cost_multiplier": 0.8,
      "codex": { "auth": "bearer", "keys": [] },
      "claude": { "auth": "bearer", "keys": [] }
    }
  },
  "keys": {}
}
```

An absent multiplier means `1`. Booleans, negative values, non-finite values,
and non-numeric values are invalid. Calculation converts the JSON value through
its decimal string representation and never uses binary floating-point money
arithmetic.

The multiplier is snapshotted into each provider selection and exact run. A
later configuration edit does not silently rewrite previously attributed cost.

## Official Price Catalog

The repository contains a normalized, versioned model-price snapshot under
`config/`. It retains earlier catalog versions when an explicit updater creates
a newer effective price version. It is generated by an explicit updater from downloadable official
OpenAI and Anthropic pricing sources. Each catalog records its source URLs,
retrieval time, schema version, currency, and effective date. Runtime summary
commands never access the network.

The first catalog includes only model IDs emitted by supported Codex and Claude
logs. Unknown model IDs retain their token counts but are reported as unpriced.
Alias handling is explicit in the catalog; the scanner does not guess by model
name prefix.

Prices are decimal USD per one million tokens and include the components
applicable to each provider:

```text
OpenAI:   input, cached input, output
Anthropic: input, output, 5-minute cache write,
           1-hour cache write, cache read
```

The updater is a development and maintenance command, not part of normal usage.
It validates the downloaded structure and required model entries before
atomically replacing the snapshot. Tests use local fixtures and never depend on
network availability.

## Cost Calculation

For a Codex usage event:

```text
uncached_input = input_tokens - cached_input_tokens

base_cost =
  uncached_input       * input_price +
  cached_input_tokens  * cached_input_price +
  output_tokens        * output_price
```

For a Claude usage event:

```text
base_cost =
  input_tokens                    * input_price +
  output_tokens                   * output_price +
  ephemeral_5m_input_tokens       * cache_write_5m_price +
  ephemeral_1h_input_tokens       * cache_write_1h_price +
  cache_read_input_tokens         * cache_read_price
```

Each price is divided by one million before summing. The provider cost is:

```text
provider_cost = base_cost * run_or_session_multiplier
```

Calculations use `decimal.Decimal`. Persisted monetary totals use integer USD
nanounits so SQLite never stores binary floating-point money. Reports format
those integer values as USD. Missing any required price component makes the
affected component unpriced; reports show priced and unpriced token counts
separately and never present a partial total as complete.

Model price lookup uses the event timestamp and the catalog effective date.
This preserves historical calculation if an official price changes later.

## Provider Selection History

After a successful `ai-provider-mode codex|claude <provider>` invocation, the
switcher records a provider selection with:

```text
timestamp, client, provider, multiplier
```

It records no token value. A successful no-op selection is still recorded
because it establishes operator intent for sessions started afterward. The
record does not reassign already running sessions.

## Exact Run Attribution

The wrapper commands are:

```text
ai-provider-run codex [client arguments...]
ai-provider-run claude [client arguments...]
```

At launch, the wrapper resolves the provider from the active client
configuration, snapshots its multiplier, records a run start, captures source
file offsets, and starts the client. It waits for that client process to exit,
then binds JSONL ranges written during that lifetime to the run and records the
run end.

For an explicit resume, the same logical session ID may receive a new run with
the provider active at resume time. Earlier runs keep their previous provider.
Reports may therefore split one logical session across providers while still
showing one logical-session identity.

Exact attribution requires the wrapper to own an unambiguous client lifetime.
If another instance of the same client is already running or overlapping writes
cannot be distinguished, the wrapper must refuse the `exact` label and mark the
affected data `estimated`. It must not silently claim precise attribution.

## File-Only Fallback Attribution

When no exact run binding exists, all events in a logical session are assigned
to the last provider selection at or before the session's first timestamp. The
selection's multiplier is used. This is intentionally session-wide even when a
provider switch occurred later.

This fallback cannot distinguish an old process that remained open across a
switch from a session that was exited and later resumed without the wrapper.
Every such result is labeled `estimated`, and text and JSON output include the
warning that session-start attribution may be inaccurate.

If no earlier provider selection exists, the complete logical session is
assigned to `historical` with multiplier `1` and attribution `historical`. It is
not assigned to the provider that happens to be active during the scan.

## SQLite Storage

The database is:

```text
~/.config/ai-provider-mode/usage.sqlite3
```

The containing directory remains private and the database and any SQLite
sidecars are created with owner-only permissions. The schema contains these
logical tables:

- `schema_meta`: database schema and parser versions.
- `source_files`: path, file identity, size, modification time, byte cursor,
  partial-line state, and last scan result.
- `provider_selections`: timestamp, client, provider, and multiplier snapshot.
- `logical_sessions`: client session ID, first and last timestamps, and source
  metadata.
- `runs`: logical session, start/end timestamps, provider, multiplier, and
  attribution method.
- `run_source_ranges` and `run_event_bindings`: exact-wrapper file ranges and
  their stable bound event keys. They preserve exact attribution through a
  rebuild without assigning later unwrapped events in the same session to an
  older run.
- `usage_events`: stable event key, run or fallback session, timestamp, model,
  normalized token components, and source location.
- `price_catalogs` and `model_prices`: imported catalog versions and normalized
  price components.

Codex events are unique by client, logical session, and turn identity. Claude
events are unique by client, logical session, and message ID. Repeated records
replace the prior normalized snapshot; they are not summed.

Normal scans seek to the stored byte cursor and parse only complete appended
JSONL lines. If a file is truncated, replaced, or rewritten before the cursor,
the scanner deletes normalized rows originating from that file and reimports
it. Moves into an archive may cause a rescan, but stable event keys prevent
double counting.

Usage events remain the calculation source of truth. Reports aggregate from
SQLite and do not reopen all JSONL files. Price catalog additions or attribution
corrections recalculate only affected database rows. Rebuild is transactional:
on scan failure it retains the prior derived data, and on success it restores
exact attribution from stable event bindings.

## Commands and Output

The usage command is:

```text
ai-provider-usage scan
ai-provider-usage summary [--from YYYY-MM-DD] [--to YYYY-MM-DD]
                          [--client codex|claude]
                          [--provider <name>|historical]
                          [--json]
ai-provider-usage sessions [same filters] [--json]
ai-provider-usage diagnose [--json]
ai-provider-usage rebuild

ai-provider-price-update [--catalog <path>]
                         [--openai-file <saved-page>]
                         [--anthropic-file <saved-page>]
```

`summary` performs an incremental scan before querying unless `--no-scan` is
provided. Its default grouping is provider, client, and model. It displays
logical session count, run count, raw token components, official base cost,
multiplier, provider cost, unpriced tokens, and counts by `exact`, `estimated`,
and `historical` attribution.

`sessions` exposes session and run breakdowns without conversation content.
`diagnose` reports scanned files, imported events, duplicate replacements,
malformed lines, unsupported records, unknown models, unpriced components, and
attribution quality. `rebuild` recreates derived usage data from local JSONL and
the provider-selection history; it does not modify source logs.

`ai-provider-price-update` is explicit maintenance work. Without file options
it downloads the configured official source pages. File options are for
offline review and tests. It validates every currently cataloged model row and
atomically replaces the catalog only after both providers parse successfully.

JSON output uses stable field names and represents money as decimal strings.
Malformed individual JSONL lines are counted and skipped. An unreadable source,
database migration failure, invalid multiplier, or invalid price catalog makes
the command fail with a path and reason but without printing source content.

## Security and Privacy

- Never store or display prompts, responses, tool arguments, tokens, or keys.
- Open JSONL files read-only and never modify Codex or Claude session files.
- Use parameterized SQLite statements and atomic config/catalog writes.
- Keep the database, provider-selection history, and backups owner-readable
  only.
- Redact paths from normal summaries; full local paths are permitted only in
  `diagnose` output requested by the user.
- The price updater is the only command in this feature allowed to use the
  network, and it runs only when explicitly invoked.

## Acceptance Criteria

1. Synthetic Codex and Claude fixtures import incrementally and idempotently.
2. Repeated Codex token-count snapshots and Claude message snapshots are not
   double counted.
3. OpenAI cached input is subtracted from regular input before pricing.
4. Claude 5-minute and 1-hour cache writes use distinct prices, with cache read
   priced separately.
5. Provider multipliers affect money only; raw tokens and official base cost do
   not change.
6. A wrapper-observed resume can assign a new provider run to an existing
   logical session while preserving earlier run attribution.
7. File-only sessions use session-start attribution and are visibly marked
   `estimated`; pre-history sessions use `historical` and multiplier `1`.
8. Unknown models and incomplete cache TTL data remain visible as unpriced and
   never produce a misleading complete total.
9. A second unchanged scan reads no already-consumed lines and leaves totals
   unchanged.
10. Tests and output contain no prompts, responses, synthetic secrets, or real
    credentials.
